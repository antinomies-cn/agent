# 项目模块验收与质量评估

本文档用于按模块定义本项目的验收标准与质量评估方式。目标不是只判断“能不能跑”，而是同时确认功能正确性、边界处理、可维护性和回归风险。

## 一、通用验收原则

1. **功能正确**：接口行为、返回结构和异常语义与设计一致。
2. **契约稳定**：对外 HTTP / WebSocket 接口尽量保持兼容，避免破坏现有调用方。
3. **边界可控**：对空值、非法值、超时、失败依赖、重复请求等场景有明确处理。
4. **安全合规**：不暴露密钥、令牌、内部地址、敏感请求体或原始隐私信息。
5. **可观测**：日志、错误码、请求 ID、trace ID、状态摘要等信息可用于定位问题。
6. **可测试**：核心逻辑尽量有单测或契约测试覆盖，回归可自动验证。
7. **可维护**：业务逻辑在 service 层，路由层保持薄，不把实现细节散落到入口层。

## 二、模块级验收与评估

### 1. `bot/app/main.py` 入口与装配层

**验收内容**

- 应用可以正常启动，`uvicorn` 可成功加载。
- 路由注册完整，`/chat`、`/ws`、`/add_urls`、`/add_urls/dry_run`、`/health`、`/qdrant/*` 等接口都按预期暴露。
- 生产环境下的路由限制生效，非允许接口返回正确的 404 语义。
- 全局异常处理器能够统一包装 HTTPException、请求校验错误和未捕获异常。
- 请求 ID / trace ID 能正确注入响应头并贯穿日志链路。

**质量评估**

- 启动时无致命错误，启动日志包含配置健康摘要。
- 异常响应结构统一，错误码、解释字段和历史兼容字段完整。
- 中间件不会泄漏 trace，上下文在请求结束后能清理。
- 入口层只负责装配，不承载复杂业务逻辑。

**建议证据**

- 启动日志截图或日志片段。
- 路由可达性检查结果。
- 异常响应样例。
- 对应回归测试通过结果。

### 2. `bot/app/api/routers/` 路由层

**验收内容**

- 每个路由只做参数接收、校验、调用 service，不堆叠业务计算。
- `/chat`、`/ws`、`/add_urls`、`/qdrant/*` 的输入输出契约稳定。
- WebSocket 路由在会话鉴权、限流和消息转发上行为正确。
- 路由层能正确映射服务层异常到统一错误响应。

**质量评估**

- 路由文件应短、清晰、职责单一。
- 新增接口时能快速定位参数来源和业务落点。
- 路由不直接访问底层存储或模型对象，避免耦合上升。

**建议证据**

- 路由契约测试。
- WebSocket 交互测试。
- 路由代码审查结果：是否存在明显业务逻辑下沉。

### 3. `bot/app/services/master_service.py` Agent 编排与记忆

**验收内容**

- `/chat` 和 `/ws` 的核心对话流程正确执行。
- 会话记忆可正常读取、写入、压缩，并在 Redis 不可用时降级到进程内内存。
- 工具选择、调用顺序、工具结果回填和最终答案生成符合预期。
- 工具调用上下文能完整传递，不能丢失 function/tool message 信息。
- 运行时对象不会被 pickle 或 deepcopy。

**质量评估**

- 对话结果在正常输入、空输入、异常输入下都有稳定输出或明确报错。
- Redis 故障不应导致整个对话失败，只应触发降级路径。
- Agent 的工具选择应尽量精准，避免无关工具被误调用。
- 核心状态管理逻辑可读、可测、可追踪。

**建议证据**

- 对话用例测试。
- Redis 可用/不可用双路径测试。
- 工具调用轨迹或日志样例。

### 4. `bot/app/llm/custom_llm.py` LLM 网关适配

**验收内容**

- OpenAI 兼容请求能正确发起，基础地址、路径和鉴权方式正确。
- 响应解析对 `choices`、`message`、`content` 缺失有防御能力。
- 非字符串内容、tool calls、function message 等情况可以安全处理。
- 网络错误、超时、限流和服务端错误有合理重试或降级处理。

**质量评估**

- 对不同网关实现兼容性强，不依赖单一返回格式。
- 出错时能返回可读、可定位的错误信息，而不是原始堆栈直接外泄。
- 适配层对 URL 规范化的处理清晰，避免 `/v1` 重复拼接。

**建议证据**

- 模拟网关返回缺字段的单元测试。
- tool call 响应解析测试。
- 真实或 mock 网关联调记录。

### 5. `bot/app/core/` 配置、网关安全与日志

**验收内容**

- 配置加载正常，环境变量缺失或非法时能给出明确提示。
- 网关安全策略生效，包括鉴权、限流、请求体大小限制和审计日志。
- CORS、生产环境限制、请求超时策略正确运行。
- 日志脱敏有效，不输出密钥、原始 token、敏感请求体等信息。

**质量评估**

- 配置健康摘要可用，且不暴露敏感值。
- 安全策略默认保守，生产环境不会默认开放高风险接口。
- 日志既能帮助排障，又不会损害隐私和安全。

**建议证据**

- 配置健康检查结果。
- 安全策略测试结果。
- 日志脱敏测试结果。

### 6. `bot/app/services/add_urls_service.py` URL 入库链路

**验收内容**

- URL 归一化、去空、合并 `url` / `urls` 参数行为正确。
- 私网、环回、链路本地地址和疑似内网主机名能够被拦截。
- 抓取、重试、退避、超时控制有效。
- 长正文会在切块前按配置截断。
- chunk 生成后的 metadata 完整，包含来源、策略、索引等信息。
- Qdrant 维度不匹配时能显式报错，避免自动误删或错误写入。

**质量评估**

- 安全性高，SSR F 风险控制明确。
- 抓取失败与局部失败可被收集，而不是直接让整个批次崩溃。
- 切块策略可解释，且对不同内容类型有可调参数。
- 维度冲突处理符合数据安全优先原则。

**建议证据**

- 安全 URL 拦截测试。
- 抓取失败与重试日志。
- dry_run 输出样例。
- Qdrant 维度冲突测试。

### 7. `bot/app/services/qdrant_service.py` 向量库运维

**验收内容**

- 集合初始化、查询、重建、健康检查接口可用。
- 本地模式与远程模式都能正常工作。
- 集合维度、距离度量和集合存在性检查准确。
- 重建操作行为清晰，避免误伤已有数据。

**质量评估**

- 运维接口返回结构统一，错误可定位。
- 集合管理逻辑稳定，不因客户端版本差异频繁失败。
- 与 embedding 配置的一致性检查明确。

**建议证据**

- 初始化 / 重建 / 健康检查测试。
- 本地与远程模式验证记录。

### 8. `bot/app/tools/` 工具层

**验收内容**

- 各工具可被 Agent 正确调用。
- 输入参数校验合理，错误信息明确。
- 外部依赖失败时不会拖垮整体对话流程。
- 返回内容可被模型消费，格式稳定。

**质量评估**

- 工具职责单一，避免工具之间相互耦合。
- 工具失败路径可预测，可降级。
- 工具日志可用于排障但不泄露敏感信息。

**建议证据**

- 工具单测。
- Agent 调用工具的集成测试。

### 9. `bot/app/schemas/` 数据模型层

**验收内容**

- 请求/响应模型与接口字段一致。
- 必填字段、默认值、枚举值、长度和类型约束清晰。
- 模型升级时对旧调用方影响可控。

**质量评估**

- Schema 清晰表达契约，减少路由层手写校验。
- 反序列化异常能被统一捕获并转成可读错误。

**建议证据**

- 模型校验测试。
- 边界值输入测试。

### 10. `bot/app/startup_check.py` 启动自检

**验收内容**

- 启动自检能识别关键配置和依赖状态。
- 自检失败时能提供明确原因。
- 不把敏感值直接打印到日志里。

**质量评估**

- 自检结果可作为上线前门禁。
- 自检逻辑稳定，不依赖不必要的外部状态。

**建议证据**

- 自检通过 / 失败案例。
- 启动日志中的健康摘要。

## 三、测试与评估方式

### 1. 功能验收

- 以接口为单位逐项验证请求和响应。
- 先验证正常流，再验证失败流、超时流、空值流和重复请求流。
- 所有关键接口至少覆盖一条成功路径和一条失败路径。

### 2. 质量评估

- **正确性**：结果是否符合预期。
- **稳定性**：在依赖抖动时是否保持可用。
- **安全性**：是否存在敏感信息泄漏或 SSRF 风险。
- **可维护性**：实现是否符合分层原则，是否容易扩展。
- **可观测性**：日志、错误码和 trace 是否足够定位问题。
- **回归风险**：修改一个模块时是否会影响已有契约。

### 3. 建议的验收证据

- 自动化测试结果。
- 关键接口的请求/响应样例。
- 失败场景日志片段。
- 配置健康检查截图或输出。
- 手工联调用例记录。

## 四、建议的验收顺序

1. 先验收入口层和全局异常处理。
2. 再验收路由层契约。
3. 然后验收 Agent、LLM 和工具链路。
4. 最后验收 URL 入库、Qdrant 和启动自检。
5. 全部通过后再做一次回归测试，确认没有破坏已有接口。

## 五、结论

本项目的验收不应只看“能跑起来”，而应同时检查分层是否清晰、异常是否统一、日志是否可追踪、安全是否可控、测试是否覆盖关键路径。按照上述模块拆分验收，可以更快定位问题，也更容易评估修改是否真的达到了可上线标准。
